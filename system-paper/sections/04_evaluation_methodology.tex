\subsection{Bottom-Up Component Evaluation}
% Why retriever first, then generator under oracle context, then end-to-end.
% Coupling problem: a weak retriever makes generator eval meaningless;
% end-to-end alone confounds retrieval quality with generation faithfulness.

\subsection{Benchmarks Used}
% mamaretrieval v0.2.0 (3,185 queries; 230,964 labeled query-chunk pairs
% from the Tier 3 top-20 x 6-retriever pool): retriever evaluation and
% oracle context for generator isolation.
% mamabench v0.2.1 (25,949 rows across 7 sources, 3 set_types: mcq /
% open_ended / open_ended_rubric): end-to-end evaluation. Construction
% details in the companion benchmarks paper [cite paper2].

\subsection{Device--Cluster Calibration}
% Most evaluation runs on cluster (llama-cpp Q4_0 on A100) rather than
% on-device (LiteRT-LM FP16 on Android GPU). Justified by a paired MCQ
% calibration on 300 rows: accuracy delta +2.7 pp (within noise),
% per-row agreement 65.0%, Cohen's kappa 0.558 -- the two runtimes are
% effectively interchangeable for aggregate scoring.
% Source: mamai-eval configs/config-v0.2.0/reports/calibration-mcq-20260519.md.

\subsection{Metric Choices}
% Retrieval: Hit Rate@k and weighted precision at k=3 (deployed depth)
% and k=20, under lenient (score >= 3) and strict (score >= 5) relevance
% thresholds on the 0--6 graded rubric [cite paper2].
% Generation under oracle: Lynx-70B PASS/FAIL rate, FAIL categorization,
% and an independent blinded calibration that converts the raw rate into
% a true-hallucination estimate.
% End-to-end: MCQ accuracy; HealthBench-style rubric scores per axis
% (judge: gpt-oss-120b, validated on a physician-labeled set); SAQ
% key-fact recall + safety verdicts.
% Latency: median per condition from repeated runs; P50/P90/P99 where
% available.
